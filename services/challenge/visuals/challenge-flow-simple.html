GHL Build Spec

Challenge — Automation Notes

1 Workflow Context
Workflow
Challenge | 7-Day Course + 13-Day Upsell (20-day total)
Trigger
Kajabi Offer Granted (7-Day Challenge) → tag challenge_purchased; if first Kajabi product, Kajabi creates user and sends its own setup-password email.
Entry
Contact has challenge_purchased AND NOT challenge_ch9_sent (prevents re-entry after full 20-day sequence).
Delivery
Kajabi course drip (Days 1-7) + GHL email sequence. Password branch on kajabi_account_created.
Purpose
Deliver the 7-day Challenge (CH-1 → CH-4) then pivot into a 13-day nurture+upsell (CH-5 → CH-9) that sets up the next step (MEM or HWN).
2 Key GHL Tags
  • challenge_purchased — primary entry flag.
  • kajabi_account_created — persistent across all Kajabi products; drives password-setup vs direct-login branch.
  • challenge_ch1_sent … challenge_ch9_sent — per-email send flags (CH-1 through CH-9) for idempotency.
  • challenge_active_d1, challenge_active_d2, … — activity heartbeats from Kajabi progress webhooks (if available).
  • challenge_inactive_d4 — set if no Day 1/2/3 activity by Day 4; routes CH-3 Midpoint → CH-3 Inactive variant.
  • challenge_completed — set on Day 7 activity or Kajabi 100%-complete webhook.
  • next_step_clicked — set on any upsell CTA click (CH-6 → CH-9); may dampen later pitches.
  • mem_purchased / hwn_active — if present, suppress upsell emails (CH-7 onward).
  • challenge_refunded, challenge_access_revoked — terminal exit flags.
3 Decision Gates
  • Steps 1-2 (Purchase + Access) — checkout fires challenge_purchased; Kajabi grants course access. Password-setup branch on kajabi_account_created.
  • CH-1 Welcome (Day 1, "Now") — primary onboarding; Kajabi login path + Day 1 expectations + bonus preview.
  • CH-2 Check-In (Day 2 · 8am) — accountability + Day 2 handoff.
  • CH-3 Midpoint (Day 4 · 8am) — active/inactive branch: IF challenge_active_d1 OR challenge_active_d2 OR challenge_active_d3 → send CH-3 Midpoint; ELSE → send CH-3 Inactive (re-engagement variant).
  • CH-4 Final Day (Day 7 · 8am) — push Day 7 completion + integration.
  • CH-5 Celebration (Day 8 · 9am) — post-completion celebration; soft positioning for next step.
  • CH-6 The Gap (Day 11 · 9am) — positioning the next step (what's still missing).
  • CH-7 Proof (Day 14 · 9am) — social proof. SUPPRESS if mem_purchased OR hwn_active.
  • CH-8 Full Picture (Day 17 · 9am) — vision/transformation. SUPPRESS if mem_purchased OR hwn_active.
  • CH-9 Objections (Day 20 · 9am) — final push / objection handling. SUPPRESS if mem_purchased OR hwn_active.
4 Exits & Errors
  • Refund — Stripe refund webhook → remove challenge_purchased, add challenge_refunded, halt workflow.
  • Upsell converted mid-flow — on mem_purchased or hwn_active, suppress CH-7 / CH-8 / CH-9 (don't pitch what they've bought).
  • Unsubscribe — GHL suppression handles delivery; workflow continues silently.
  • Course access revoked — Kajabi access-removed webhook → add challenge_access_revoked, halt workflow.
  • Fallback (no activity webhook) — if Kajabi progress webhooks aren't wired, collapse the CH-3 active/inactive branch to a time-only split (send Midpoint to everyone, no Inactive variant).
5 Open Questions
Upsell target — MEM or HWN?
CH-5 → CH-9 are framed generically as "the next step". Confirm whether the pitch sequence targets MEM ($1,497 headline), HWN ($47/mo or $470/yr), or split by segment (e.g., high-intent → MEM, low-intent → HWN).
Kajabi → GHL activity heartbeat.
Does Kajabi push lesson-completed webhooks to GHL? The CH-3 active/inactive branch depends on this. If no, fall back to time-only split.
Refund / cancel exits — pattern not yet canonical.
Need shared convention across UHA / Challenge / MEM / HWN. Current proposal: Stripe refund → remove entry tag + add *_refunded + halt.
9-step go/no-go test — pending from prior sessions.
Build the validation checklist before go-live (trigger fires, password branch, daily course sends align, active/inactive branch works, upsell suppression works, refund exits cleanly, all 9 tags idempotent).

Grant Challenge Offer — Zap Setup
Fires on every Challenge purchase. Creates/updates the Kajabi contact and grants the offer. Kajabi handles the setup email for new users automatically.
1
Zap trigger — Webhooks by Zapier · Catch Hook
Copy the catch-hook URL. In GHL, add a Webhook action to the workflow that fires when the purchased_challenge tag is added to a contact. POST contact fields: email, first_name, last_name, contact_id, phone.
2
Map GHL payload → Zap sample
Run the GHL workflow once with a test contact so Zapier pulls a real sample. Confirm all five fields appear in the sample before moving on — missing fields here break every downstream step.
3
Kajabi · Find or Create Person
Action: Find Person with Create if not found = true. Lookup by email. Output exposes person_id plus a flag for new vs existing — this is the new/returning user branch.
4
Kajabi · Grant an Offer
Action: Grant an Offer. Person = step 3 person_id. Offer = Challenge. On new users, Kajabi auto-sends the Set Up Your Account email — we don't build that email. Returning users skip it and keep existing login.
5
GHL · Write back the Kajabi tag
Action: GHL Update Contact → add tag kajabi_account_created. This is how every future Zap (MEM, HWN, upsells) knows the user already has a Kajabi login and skips the setup email.
6
Publish + test end-to-end
Turn the Zap ON. Run one real $1 test purchase. Verify: contact exists in Kajabi · Challenge offer granted · new user got the setup email (returning user did not) · kajabi_account_created tag present in GHL.
