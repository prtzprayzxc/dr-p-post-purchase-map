MEM — Automation Notes

1 Workflow Context
Workflow
MEM | 12-Week Course + Graduation → HWN Cross-Sell
Trigger
Kajabi Offer Granted (MEM or MEM Payment Plan) → tag mem_purchased; password branch on kajabi_account_created.
Entry
Contact has mem_purchased AND NOT mem_14_sent (prevents re-entry after full sequence).
Delivery
Kajabi weekly drip + GHL email sequence (14 emails) + community invite + live-call calendar. Payment-plan path branches into a parallel dunning sub-flow (MEM-D1 → MEM-D3 → MEM-DR).
Price
$1,497 headline one-time (with DOCTORP500) OR payment plan — 3 × $667/mo or $497 + 10 × $150. Payment-plan path triggers dunning sub-flow if any installment fails.
Purpose
Deliver 12-week Mastermind curriculum (MEM-1 → MEM-10) then pivot to HWN cross-sell (MEM-11 → MEM-14). Cross-sell emails (MEM-12/13/14) skip if purchased_hwn already exists.
2 Key GHL Tags
  • mem_purchased — primary entry flag.
  • mem_payment_plan — set if payment-plan offer; gates dunning sub-flow.
  • kajabi_account_created — persistent; drives password-setup vs direct-login branch.
  • mem_1_sent … mem_14_sent — per-email send flags (MEM-1 Welcome through MEM-14 Continue HWN).
  • mem_d1_sent, mem_d2_sent, mem_d3_sent, mem_dr_sent — dunning sub-flow send flags.
  • mem_community_joined — from community platform webhook.
  • mem_payment_failed, mem_payment_recovered — dunning state.
  • purchased_hwn — suppresses MEM-12 / MEM-13 / MEM-14 (don't cross-sell HWN to existing members).
  • mem_graduated — set at MEM-10 send or on Kajabi course-complete webhook.
  • mem_paused, mem_cancelled, mem_refunded — terminal/pause states.
3 Decision Gates
  • Steps 1-2 (Purchase + Access) — Kajabi grants access; password-setup branch on kajabi_account_created; community + first-call calendar links delivered.
  • MEM-1 Welcome (Now) — onboarding moment; community + first-call calendar + password path.
  • MEM-2 Rhythm (Day 2 · 9am) — set expectations for weekly cadence.
  • MEM-3 Accountability (Day 4 · 9am) — community-intro prompt.
  • MEM-4 Nutrition (Week 2 · Mon) — weekly module 1.
  • MEM-5 Movement (Week 3 · Mon) — weekly module 2.
  • MEM-6 Hormones (Week 4 · Mon) — weekly module 3.
  • MEM-7 Midpoint (Week 6 · Mon) — halfway check-in.
  • MEM-8 Final Stretch (Week 9 · Mon) — Q4 push.
  • MEM-9 Final Module (Week 12 · Mon) — last content module.
  • MEM-10 Graduation (Week 12 + 3d) — celebration; sets mem_graduated.
  • MEM-11 What Now? (Bonus final stretch) — transition email that pivots to HWN as the next step.
  • MEM-12 HWN Reminder (-14 days) — first HWN cross-sell. SKIP if purchased_hwn.
  • MEM-13 HWN Reminder (-7 days) — second HWN push. SKIP if purchased_hwn.
  • MEM-14 Continue HWN (-2 days) — final cross-sell. SKIP if purchased_hwn.
  • Dunning sub-flow (payment-plan only) — MEM-D1 +2h → MEM-D2 Day 3 → MEM-D3 Day 6; MEM-DR fires on recovery. Pauses main cadence while active.
4 Exits & Errors
  • Refund — Stripe refund webhook → remove mem_purchased, add mem_refunded, halt.
  • Cancel — manual or Kajabi cancel → add mem_cancelled, halt all future weekly + cross-sell sends.
  • Payment pause — add mem_paused, pause weekly cadence; resume on mem_payment_recovered.
  • HWN purchased mid-flow — on purchased_hwn, suppress MEM-12 / MEM-13 / MEM-14.
  • Dunning exhausted — if MEM-D3 fires without recovery within grace period → add mem_cancelled, halt.
  • Unsubscribe — suppression list; workflow continues silently.

Dunning ↔ main-cadence interaction.
When dunning is active, do we pause MEM-n weekly sends, or keep them flowing? Current assumption: pause until MEM-DR recovery. Confirm.
Refund / cancel exits — pattern not yet canonical.
Align with UHA / Challenge / HWN on a shared exit convention.

Setup One-time Zap build · GHL → Zapier → Kajabi
Grant MEM + Community Offers — Zap Setup
Fires on every MEM purchase. Creates or updates the Kajabi person, grants the core MEM offer plus community access, and preserves the no-new-password branch for returning Kajabi users.
1
Zap trigger — catch the MEM purchase webhook
From the MEM purchase workflow in GHL, POST the buyer payload into Zapier. Include at minimum email, first_name, last_name, contact_id, plan type, and any payment-status fields used for dunning branches.
2
Pull a live sample and confirm plan metadata
Run one test purchase so Zapier sees the real payload. Confirm the sample distinguishes full-pay vs payment-plan buyers because MEM-1 variants and downstream recovery logic depend on that state.
3
Kajabi — find or create the person
Use Find Person with create-if-missing enabled, keyed on email. That output becomes the single person record used for the MEM course, the Empowered Sister community, and any bundled access.
4
Grant the MEM offer and community access
Grant the MEM course offer, then grant the community product/access in the same Zap. If the HWN bonus is still bundled at purchase, include that offer grant here too so onboarding matches the confirmation page promise.
5
Write back Kajabi-account state into GHL
After a successful grant, update the GHL contact with kajabi_account_created. Future Kajabi-based flows use that tag to suppress the automatic setup email for returning users and send direct-login messaging instead.
6
Test new-user and returning-user paths
Verify one brand-new buyer gets the Kajabi setup email, one existing Kajabi user skips it, both land in MEM + community correctly, and the GHL contact reflects the right tags for weekly cadence and dunning follow-up.
